Kafka cluster:(one+ server in 1+ datacenters)
  • stores streams of records in categories called topics
  • Each record consists of:
TOPIC and partitions
  • Category or feed name to which records are published
  • producer (1 ←→ 1) topic (1 ←→ 0...N) subscribers

topic 1 ←→ (partitioned) log  (Immutable sequence of records)

   Partitions serve several purposes:
    - allow scaling beyond a single server
      (individual partition must fit on the host server)
    - act as the unit of parallelism
  • Retention policy indicates how much time the records will be available for compsumption before being discarded.
  • Note: By configuring which topics can produce or consume data multi-tenancy can be enabled
Partition replication
  • Each partition is replicated across a configurable number of servers for fault tolerance
  • Kafka allows producers to wait on acknowledgement so that a write isn't considered complete until it is fully replicated and guaranteed to persist even if the server written to fails
  • Each partition has one server which acts as the "leader" and zero or more servers which act as "followers". The leader handles all read and write requests for the partition while the followers passively replicate the leader. If the leader fails, one of the followers will automatically become the new leader. Each server acts as a leader for some of its partitions and a follower for others so load is well balanced within the cluster.
  • For a topic with replication factor N, we will tolerate up to N-1 server failures without losing any records committed to the log.
  • Kafka MirrorMaker provides geo-replication support for your clusters
Consumer Groups:
(load-balance/scale/broadcast) 
Log partitions are (dnyamically) divided over consumer instances so that each instance
is the exclusive consumer of a "fair share" of partitions at any point in time.
                      
                      ┌──────────┬────────────────┐
LOG                   │ CONSUMER │   CONSUMER     │ 
                      │ GROUP A  │   GROUP B      │ Each consumer group is just a cluster 
                      │ C-1  C-2 │C-3 C-4 C-5 C-6 │ of consumers instead of a single process 
         ┌────────────┼──────────┼────────────────┤ in the publish-subscribe model
SERVER1  │Partition-0 │  X       │X               │ 
         │Partition-3 │  X       │     X          │ Ussually there is 1 to 1 relation between a
         ├────────────┼──────────┼────────────────┤ consumer group and a "logical subscriber"
SERVER2  │Partition-1 │      X   │         X      │
         │Partition-2 │      X   │             X  │
         └────────────┴──────────┴────────────────┘
                       * Consumer group instances
                         must be smaller or equal to
                         number of partitions

The consumer group generalizes the queue and publish-subscribe:
  - As with a queue the consumer group allows you to divide (scale) up
    processing over a collection of processes (the members of the consumer group). 
  - As with publish-subscribe, Kafka allows you to broadcast messages to
    multiple consumer groups
About Total Ordering
  • Kafka only provides a total order over records within a partition, not between different partitions in a topic. Per-partition ordering combined with the ability to partition data by key is sufficient for most applications. However, if you require a total order over records this can be achieved with a topic that has only one partition, though this will mean only one consumer process per consumer group.
  • Messages sent by a producer to a particular topic partition are guaranteed to be appended in the order they are sent.

Producer API
send streams of data to topics in the cluster 		Consumer API Streams API (more info)
  • allows building applications that do non-trivial processing that compute aggregations off of streams or join streams together.
  • The streams API builds on the core primitives Kafka provides: it uses the producer and consumer APIs for input, uses Kafka for stateful storage, and uses the same group mechanism for fault tolerance among the stream processor instances.
Connect API (more info)
  • allows building and running reusable producers or consumers that connect Kafka topics to existing applications or data systems. For example, a connector to a relational database might capture every change to a table

Quick set-up: 
# STEP 1: Download from here
~ tar -xzf kafka_2.11-1.1.0.tgz ; cd kafka_2.11-1.1.0

# STEP 2: Start the server
# STEP 2.1: Start Zookeeper
~ bin/zookeeper-server-start.sh config/zookeeper.properties 1>zookeeper.log 2>&1 &
# STEP 2.2: Start Kafka server:
~ bin/kafka-server-start.sh config/server.properties 1>kafka.log 2>&1 &

# STEP 3: Create a topic named "test":
~ bin/kafka-topics.sh --create --zookeeper localhost:2181 --replication-factor 1 --partitions 1 --topic test
# STEP 3.1: Check created topic:
~ bin/kafka-topics.sh --list --zookeeper localhost:2181
(ouput)
test

# STEP 3 (Alt): instead of manually creating topics, brokers can be configured
#              to auto-create topics when publishing to non-existent ones

# STEP 4: Send some messages
~ bin/kafka-console-producer.sh --broker-list      localhost:9092 --topic test
This is a message
This is another message
Ctrl+V

# STEP 5: Start a consumer
~ bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic test --from-beginning
This is a message
This is another message

# STEP 6: expanding our cluster to three brokers/nodes in our local machine:
# STEP 6.1: Add 2 new broker configs.
~ cp config/server.properties config/server-1.properties # 
~ cp config/server.properties config/server-2.properties #

config/server-1.properties:        config/server-2.properties:
    broker.id=1                        broker.id=2                 ← unique permanent id for each node
    listeners=PLAINTEXT://:9093        listeners=PLAINTEXT://:9094
    log.dir=/tmp/kafka-logs-1          log.dir=/tmp/kafka-logs-2   ← avoid overwrite 
# STEP 6.2: startup the kafka servers:
~ bin/kafka-server-start.sh config/server-1.properties ... 
~ bin/kafka-server-start.sh config/server-2.properties ... 

# STEP 6.3: create a new topic with a replication factor of three:
~ bin/kafka-topics.sh --create --zookeeper localhost:2181 --replication-factor 3 --partitions 1 --topic my-replicated-topic

# STEP 6.4: Check know which broker is doing what
~ bin/kafka-topics.sh --describe --zookeeper localhost:2181 --topic my-replicated-topic
Topic:my-replicated-topic   PartitionCount:1    ReplicationFactor:3 Configs:              ← summary of all the partitions
    Topic: my-replicated-topic  Partition: 0    Leader: 1   Replicas: 1,2,0 Isr: 1,2,0    ←info about a given partition

    "leader"  : node responsible for all reads/writes for the given partition
                If "leader" is killed an slave will take the roll.
    "replicas": node list replicating the log for this partition (regardless of leadership or alive-status)
    "isr"     : set of "in-sync" replicas. (subset of "replicas" currently alive and in sync with "leader")
